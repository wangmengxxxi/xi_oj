一、方案概述

1.1 项目背景

本方案基于现有XI OJ平台进行二次开发与优化，在保留原有核心判题、题库、用户体系的基础上，深度整合AIGC能力，解决用户编程学习中「错题无指导、解题无思路、学习无路径」的核心痛点，同时完善平台的用户管理、互动社区、个性化配置等基础功能，打造「练-评-学-练」闭环的智能化OJ学习平台。

1.2 核心目标

  1. 落地AIGC核心能力：基于LangChain4j实现RAG+Agent技术架构，完成AI代码分析、AI智能判题、AI问答助手、AI题目解析、相似题推荐、AI错题本六大核心AI功能；
  2. 完善平台基础能力：新增AI功能全局开关、用户信息管理、登录注册优化、题目评论区等功能，补齐平台产品能力；
  3. 前端体验优化：对齐现有AI分析、AI问答页面的交互逻辑，完成前端页面的统一优化与适配；
  4. 可拓展架构设计：实现业务层与AIGC层解耦，支持后续功能的快速迭代。

1.3 适配范围

  • 完全兼容现有OJ平台的Java SpringBoot技术栈、MySQL数据库、代码沙箱判题体系；
  • 完全适配「AI判题、AI助手自动回复、AI分析题目、AI分析提交代码、AI问答、AI开关、用户信息管理、评论区、AI错题本」的功能需求；
  • 支持Java 8及以上版本，兼容LangChain4j全版本核心能力。

二、整体架构设计

本方案采用分层解耦架构，将AIGC能力作为独立的核心层封装，与现有业务系统完全解耦，既保证现有功能的稳定性，又支持后续能力的快速迭代。

2.2 分层职责说明

层级 核心模块 职责说明
前端展示层 AI分析页面、AI问答页面、题目详情页、代码提交页、用户中心、评论区、错题本 负责用户交互与页面渲染，对接后端AI接口与业务接口，对齐现有页面的交互逻辑
接口网关层 接口鉴权、流量控制、AI开关拦截、参数校验 统一入口管理，拦截非法请求，根据AI全局开关控制AI接口的访问，保障接口安全
业务服务层 题库管理、代码判题、用户管理、评论管理、错题管理、提交记录管理、系统配置管理 保留OJ平台原有核心业务能力，新增用户管理、评论区、错题本等扩展功能，为AI层提供业务数据支撑
AIGC核心层 RAG检索模块、Agent执行模块、大模型适配模块、工具调用模块 整个平台的AI能力核心，封装所有AI相关逻辑，与业务层完全解耦，通过工具调用对接业务能力
数据存储层 MySQL业务库、Milvus向量库、Redis缓存库 分别存储业务数据、AI向量知识库、高频缓存数据（AI问答、相似题检索结果）
基础设施层 代码沙箱、大模型API服务、日志监控、消息队列 提供底层能力支撑，包括代码运行、大模型调用、系统监控、异步任务处理

2.3 AIGC核心层内部架构

  1. flowchart LR
  2.     A[请求入口] --> B[Agent执行引擎]
  3.     B --> C[RAG检索模块]
  4.     B --> D[工具调用模块]
  5.     C --> E[向量数据库]
  6.     D --> F[OJ业务接口]

  • Agent执行引擎：基于LangChain4j的AgenticServices实现，负责AI的思考、工具调用决策、结果整合，是AIGC层的调度核心；
  • RAG检索模块：负责题目、题解、知识点、错题分析的向量检索，为AI提供精准的上下文信息，解决大模型幻觉问题；
  • 工具调用模块：封装OJ平台的题库查询、代码判题、用户数据查询、错题查询等能力，供Agent按需调用；
  • 大模型适配层：统一封装大模型调用接口，支持多模型快速切换，兼容国内主流大模型。

4.2.1 AI系统配置表（ai_config）

用于管理AI功能全局开关、大模型配置、限流规则等，实现AI功能的一键启停与动态配置。

  1. CREATE TABLE IF NOT EXISTS ai_config
  2. (
  3.     id          bigint auto_increment comment 'id' primary key,
  4.     config_key  varchar(128) NOT NULL comment '配置键',
  5.     config_value text comment '配置值',
  6.     description varchar(512) comment '配置描述',
  7.     is_enable   tinyint default 1 not null comment '是否启用',
  8.     createTime  datetime default CURRENT_TIMESTAMP not null comment '创建时间',
  9.     updateTime  datetime default CURRENT_TIMESTAMP not null on update CURRENT_TIMESTAMP comment '更新时间',
  10.     UNIQUE KEY uk_config_key (config_key)
  11. ) comment 'AI系统配置表' collate = utf8mb4_unicode_ci;

初始化核心配置：

ai.model.api_key 已移除，API Key 属于敏感凭证，通过环境变量注入，不落库，详见下方说明。

config_key config_value description
ai.global.enable true AI功能全局开关
ai.model.base_url https://dashscope.aliyuncs.com/compatible-mode/v1 百炼OpenAI兼容端点，通常无需修改
ai.model.name qwen-plus 聊天模型名称（可选：qwen-turbo / qwen-plus / qwen-max）
ai.model.embedding_name text-embedding-v3 嵌入模型名称，修改后需重建向量索引
ai.rag.top_k 3 RAG检索返回条数（建议3-5）
ai.rag.similarity_threshold 0.7 RAG最小相似度阈值（0-1，值越高检索越严格）

API Key 配置方式（环境变量注入）：

  1. # application.yml
  2. ai:
  3.   model:
  4.     api-key: ${AI_API_KEY}   # 从环境变量读取，不写入代码和数据库

部署时在服务器或 Docker 中设置环境变量：

  1. # Linux / Docker
  2. export AI_API_KEY=sk-xxxxxxxxxxxxxxxx

5.1.1 RAG检索模块

核心职责：为AI提供精准的上下文信息，解决大模型幻觉问题，保证回答的准确性。
核心流程：

  1. flowchart TD
  2.     A[用户问题/题目信息/错题信息] --> B[文本预处理]
  3.     B --> C[生成问题向量]
  4.     C --> D[向量库检索]
  5.     D --> E[相似度过滤（阈值≥0.7）]
  6.     E --> F[上下文压缩与重排序]
  7.     F --> G[返回给Agent的精准上下文]

核心代码实现：

  1. /**
  2.  * RAG检索核心类，负责向量库检索与上下文处理
  3.  */
  4. @Component
  5. public class OJKnowledgeRetriever {
  7.     @Autowired
  8.     private MilvusEmbeddingStore embeddingStore;
  9.     @Autowired
  10.     private QwenEmbeddingModel embeddingModel;
  12.     /**
  13.      * 核心检索方法
  14.      * @param query 用户问题/题目关键词
  15.      * @param topK 返回条数
  16.      * @param minScore 最小相似度阈值
  17.      * @return 检索到的上下文内容
  18.      */
  19.     public String retrieve(String query, int topK, double minScore) {
  20.         // 1. 生成问题向量
  21.         Embedding queryEmbedding = embeddingModel.embed(query).content();
  23.         // 2. 向量库检索
  24.         List<EmbeddingMatch<TextSegment>> matches = embeddingStore.findRelevant(queryEmbedding, topK);
  26.         // 3. 相似度过滤与内容拼接
  27.         String context = matches.stream()
  28.                 .filter(match -> match.score() >= minScore)
  29.                 .map(EmbeddingMatch::embeddedObject)
  30.                 .map(TextSegment::text)
  31.                 .collect(Collectors.joining("\n\n"));
  33.         // 4. 兜底返回
  34.         return context.isBlank() ? "无相关知识点" : context;
  35.     }
  37.     /**
  38.      * 相似题检索方法
  39.      * @param questionId 题目id
  40.      * @param questionContent 题目内容
  41.      * @return 相似题目id列表
  42.      */
  43.     public List<Long> retrieveSimilarQuestion(Long questionId, String questionContent) {
  44.         Embedding queryEmbedding = embeddingModel.embed(questionContent).content();
  45.         return embeddingStore.findRelevant(queryEmbedding, 4)
  46.                 .stream()
  47.                 .filter(match -> match.score() >= 0.75)
  48.                 .map(EmbeddingMatch::embeddedObject)
  49.                 .map(segment -> segment.metadata().getLong("question_id"))
  50.                 .filter(id -> !id.equals(questionId))
  51.                 .collect(Collectors.toList());
  52.     }
  54.     /**
  55.      * 按 content_type 过滤的检索方法
  56.      * 供 5.2 代码分析、5.5 错题分析等无状态模块手动调用，精准控制检索范围
  57.      * @param query       检索关键词（题目标题+考点拼接）
  58.      * @param contentTypes 内容类型过滤，逗号分隔（如 "代码模板,错题分析"）
  59.      * @param topK        返回条数
  60.      * @param minScore    最小相似度阈值
  61.      * @return 过滤后的上下文内容
  62.      */
  63.     public String retrieveByType(String query, String contentTypes, int topK, double minScore) {
  64.         Embedding queryEmbedding = embeddingModel.embed(query).content();
  65.         List<String> typeList = Arrays.asList(contentTypes.split(","));
  66.         // 多取一倍以保证过滤后仍有足够结果
  67.         List<EmbeddingMatch<TextSegment>> matches = embeddingStore.findRelevant(queryEmbedding, topK * 2);
  68.         String context = matches.stream()
  69.                 .filter(match -> match.score() >= minScore)
  70.                 .filter(match -> typeList.contains(
  71.                         match.embeddedObject().metadata().getString("content_type")))
  72.                 .limit(topK)
  73.                 .map(EmbeddingMatch::embeddedObject)
  74.                 .map(TextSegment::text)
  75.                 .collect(Collectors.joining("\n\n"));
  76.         return context.isBlank() ? "无相关知识点" : context;
  77.     }
  78. }

Service 层实现（含 SSE 结束后异步持久化）：

关键设计：Reactor 的 doOnComplete 回调在最后一个 token 推送完毕后触发，此时用 @Async 方法异步写库，不阻塞 SSE 推流线程。同时提供历史记录查询和清空会话接口。

  1. @Service
  2. @Slf4j
  3. public class AiChatService {
  5.     @Autowired
  6.     private OJChatAgent ojChatAgent;
  7.     @Autowired
  8.     private AiChatRecordMapper chatRecordMapper;
  9.     @Autowired
  10.     private Cache<String, ChatMemory> chatMemoryCache; // 注入工厂中的 Caffeine 缓存
  12.     /**
  13.      * 非流式问答 + 同步持久化
  14.      */
  15.     public String chat(String chatId, Long userId, String message) {
  16.         String answer = ojChatAgent.chat(chatId, message);
  17.         saveRecord(userId, chatId, message, answer);
  18.         return answer;
  19.     }
  21.     /**
  22.      * SSE 流式问答 + 异步持久化
  23.      * doOnNext：每个 token 追加到 buffer；
  24.      * doOnComplete：流结束后用 @Async 方法异步写库，不阻塞推流线程。
  25.      */
  26.     public Flux<String> chatStream(String chatId, Long userId, String message) {
  27.         StringBuilder buffer = new StringBuilder();
  28.         return ojChatAgent.chatStream(chatId, message)
  29.                 .doOnNext(buffer::append)
  30.                 .doOnComplete(() -> saveRecordAsync(userId, chatId, message, buffer.toString()))
  31.                 .doOnError(e -> log.error("[AI问答] 流式异常 chatId={}: {}", chatId, e.getMessage()));
  32.     }
  34.     /** 同步写库（非流式场景） */
  35.     private void saveRecord(Long userId, String chatId, String question, String answer) {
  36.         AiChatRecord record = new AiChatRecord();
  37.         record.setUserId(userId);
  38.         record.setChatId(chatId);
  39.         record.setQuestion(question);
  40.         record.setAnswer(answer);
  41.         chatRecordMapper.insert(record);
  42.     }
  44.     /**
  45.      * 异步写库（流式场景，独立线程不阻塞 Reactor 推流线程）
  46.      * 依赖主类或配置类上的 @EnableAsync
  47.      */
  48.     @Async
  49.     public void saveRecordAsync(Long userId, String chatId, String question, String answer) {

  51.             saveRecord(userId, chatId, question, answer);
  52.         } catch (Exception e) {
  53.             log.error("[AI问答] 对话记录写库失败 chatId={}: {}", chatId, e.getMessage());
  54.         }
  55.     }
  57.     /** 查询用户某会话的历史记录 */
  58.     public List<AiChatRecord> getChatHistory(Long userId, String chatId) {
  59.         return chatRecordMapper.selectByUserAndChat(userId, chatId);
  60.     }

5.5 AI错题本模块

功能描述：自动收集用户错题，AI分析错误原因，生成针对性的复习计划与同类题目推荐，帮助用户查漏补缺，巩固知识点。

调用模式：无状态单次调用，直接注入共享 ChatModel，RAG 检索由 Service 层手动执行后注入 Prompt（与 5.2 相同模式）。

核心流程：

  1. 错题自动收集：用户提交代码判题失败（WA/TLE/RE等）后，系统自动将「题目ID、用户ID、错误代码、判题结果」存入ai_wrong_question表；
  2. AI错误分析：Service 层手动调用 OJKnowledgeRetriever.retrieveByType() 检索「错题分析」类知识点，将结果与题目信息、错误代码拼装为完整 Prompt，直接调用 ChatModel；
  3. 复习计划生成：Agent根据用户的错误类型、题目难度、考点，结合艾宾浩斯遗忘曲线，生成个性化的复习计划，包括下次复习时间、复习重点；
  4. 同类题目推荐：通过RAG检索与当前错题考点、难度、错误类型相似的3-4道题，存入similar_questions字段；
  5. 复习提醒与记录：用户完成复习后，更新is_reviewed、review_count、next_review_time字段，系统根据next_review_time推送复习提醒。

5.5.1 错题自动收集触发点

关键说明：错题收集必须在原有判题结果处理逻辑中埋点，不能由用户手动触发。在现有 JudgeService 或判题结果回调处添加如下逻辑。

WrongQuestionCollector.java（独立 Component，方便注入任意位置）：

  1. /**
  2.  * 错题自动收集器
  3.  * 调用位置：JudgeService 判题完成后、判题结果消费者（MQ消费端）均可注入调用
  4.  */
  5. @Component
  6. @Slf4j
  7. public class WrongQuestionCollector {
  9.     @Autowired
  10.     private AiWrongQuestionMapper wrongQuestionMapper;
  12.     /**
  13.      * 判题结果回调入口（非 AC 结果自动入错题本）
  14.      * 幂等处理：同一用户同一题已有记录则更新（清空旧分析），否则新建
  15.      *
  16.      * @param userId      提交用户ID
  17.      * @param questionId  题目ID
  18.      * @param code        用户提交代码
  19.      * @param language    代码语言
  20.      * @param judgeResult 判题结果 DTO
  21.      */
  22.     public void collect(Long userId, Long questionId, String code,
  23.                         String language, JudgeResultDTO judgeResult) {
  24.         String status = judgeResult.getStatus();
  25.         // AC 不收入错题本
  26.         if ("Accepted".equalsIgnoreCase(status)) {
  27.             return;
  28.         }
  29.         try {
  30.             AiWrongQuestion existing = wrongQuestionMapper.selectByUserAndQuestion(userId, questionId);
  31.             if (existing != null) {
  32.                 // 更新：清空旧分析，重置复习状态，准备重新分析
  33.                 existing.setWrongCode(code);
  34.                 existing.setWrongJudgeResult(status);
  35.                 existing.setWrongAnalysis(null);
  36.                 existing.setReviewPlan(null);
  37.                 existing.setIsReviewed(0);
  38.                 wrongQuestionMapper.updateById(existing);
  39.                 log.info("[错题收集] 更新错题记录 userId={} questionId={} status={}", userId, questionId, status);
  40.             } else {
  41.                 // 新增
  42.                 AiWrongQuestion wrong = new AiWrongQuestion();
  43.                 wrong.setUserId(userId);
  44.                 wrong.setQuestionId(questionId);
  45.                 wrong.setWrongCode(code);
  46.                 wrong.setWrongJudgeResult(status);
  47.                 wrongQuestionMapper.insert(wrong);
  48.                 log.info("[错题收集] 新增错题记录 userId={} questionId={} status={}", userId, questionId, status);
  49.             }
  50.         } catch (Exception e) {
  51.             // 错题收集失败不影响主判题流程
  52.             log.error("[错题收集] 写库失败 userId={} questionId={}: {}", userId, questionId, e.getMessage());
  53.         }
  54.     }
  55. }

  24.      * 获取题目评论树（根评论按点赞数倒序，子评论挂载到父评论下）
  25.      */
  26.     public List<CommentVO> getCommentTree(Long questionId) {
  27.         List<QuestionComment> all = commentMapper.selectByQuestionId(questionId);
  28.         Map<Long, CommentVO> map = new LinkedHashMap<>();
  29.         List<CommentVO> roots = new ArrayList<>();
  30.         for (QuestionComment c : all) {
  31.             CommentVO vo = CommentVO.from(c);
  32.             map.put(c.getId(), vo);
  33.             if (c.getParentId() == 0) roots.add(vo);
  34.         }
  35.         for (QuestionComment c : all) {
  36.             if (c.getParentId() != 0 && map.containsKey(c.getParentId())) {
  37.                 map.get(c.getParentId()).getReplies().add(map.get(c.getId()));
  38.             }
  39.         }
  40.         // 热门评论优先
  41.         roots.sort(Comparator.comparingInt(CommentVO::getLikeNum).reversed());
  42.         return roots;
  43.     }
  45.     /**
  46.      * 点赞 / 取消点赞（Redis Set 防重复点赞）
  47.      * @return true=点赞成功，false=取消点赞
  48.      */
  49.     public boolean toggleLike(Long commentId, Long userId) {
  50.         String likeKey = "comment:like:" + commentId;
  51.         String userStr = String.valueOf(userId);
  52.         Boolean liked = redisTemplate.opsForSet().isMember(likeKey, userStr);
  53.         if (Boolean.TRUE.equals(liked)) {
  54.             redisTemplate.opsForSet().remove(likeKey, userStr);
  55.             commentMapper.decrementLike(commentId);
  56.             return false;
  57.         } else {
  58.             redisTemplate.opsForSet().add(likeKey, userStr);
  59.             commentMapper.incrementLike(commentId);
  60.             return true;
  61.         }
  62.     }
  64.     /**
  65.      * 删除评论（逻辑删除，仅作者或管理员可操作）
  66.      */
  67.     public void deleteComment(Long commentId, Long userId, boolean isAdmin) {
  68.         QuestionComment comment = commentMapper.selectById(commentId);
  69.         if (comment == null) throw new BusinessException(ErrorCode.NOT_FOUND_ERROR, "评论不存在");
  70.         if (!isAdmin && !comment.getUserId().equals(userId)) {
  71.             throw new BusinessException(ErrorCode.NO_AUTH_ERROR, "无权限删除该评论");
  72.         }
  73.         comment.setIsDelete(1);
  74.         commentMapper.updateById(comment);
  75.     }

八、风险控制与优化方案

8.1 核心风险与应对方案

风险场景 风险描述 应对方案
大模型API异常 API调用超时、限流、服务不可用 1. 实现多模型降级切换，主模型异常时自动切换备用模型；2. 接口超时重试机制；3. 高频问题缓存，减少API调用
AI回答幻觉 AI生成错误的解题思路、代码分析、错题分析 1. 强化RAG检索，所有回答必须基于检索到的知识点；2. Prompt中增加约束，禁止编造题目、考点；3. 增加回答结果校验，与题目信息、判题结果交叉验证
并发量过高 大量用户同时调用AI接口，导致服务压力过大 1. 用户级别的调用次数限流；2. AI任务异步处理，避免阻塞主线程；3. 接口熔断降级，服务压力过大时关闭非核心AI功能
向量库检索性能下降 随着题目数据增多，检索耗时增加 1. Milvus开启索引优化，提升检索速度；2. 按考点、难度对向量库分库分表；3. 高频检索结果缓存到Redis

8.2 性能优化方案

  1. 缓存优化：对高频题目解析、相似题检索结果、AI问答内容、错题分析进行Redis缓存，缓存有效期1小时，减少重复计算与API调用；
  2. 异步优化：代码分析、错题分析、向量库数据同步等耗时操作，通过消息队列异步处理，不阻塞用户主流程；
  3. RAG优化：采用「关键词检索+向量检索」的混合检索模式，提升检索精准度与速度；对知识点精细化分割，减少检索噪声；
  4. 大模型优化：调整大模型参数，temperature设置为0.1-0.3，减少随机性；maxTokens按需设置，避免无效token消耗，提升响应速度。
